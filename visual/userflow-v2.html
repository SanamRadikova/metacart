Архитектура логики иерархии,Graceful Degradation и пересчета профилей (Июнь 2026)
Этап 1: Входной слой (Ввод данных)
👤 Авторизация, Согласие IRB и Сбор данных
Пользователь подписывает информированное согласие на участие в клиническом пилоте $N=40$. Заполняет профиль: Пол, Возраст (критерий включения 18–65 лет), культурную группу и гормональные модификаторы.

Доступные каналы поступления данных:
Уровень А: Лабораторные анализы (PDF/Ручной ввод) Уровень B: Потоки Apple Health / Google Fit (HRV, Пульс) Уровень C: Гормональный статус (Сдвиг порогов в лютеиновой фазе)
⚙️ Проверка полноты данных (Graceful Degradation — Уровень D):
Все ли обязательные маркеры Уровня А (Глюкоза натощак, HbA1c, TG, HDL) присутствуют в БД?
Частичные данные (Деградация)
Движок на Go не блокирует расчет. Недостающие оси помечаются статусом NO_DATA и исключаются из сита иерархии.
  • Нет липидов (TG/HDL) → Исключаем Ось 2.
  • Нет hs-CRP → Исключаем Ось 3.
  • Нет TSH → Исключаем Ось 4.
  • Доступны только базовые профили: Profile 1 и Profile 2.
Полные данные (Штатный режим)
Все 5 метаболических осей активированы и оцениваются по референсам:
  • Расчет жестких маркеров крови по референсам.
  • Парсинг фонового HRV из Apple Health (Ось 5).
  • Динамическая подстройка порогов под лютеиновую фазу (сдвиг нормы на 20%).
Этап 2: Осевой Анализ на бэкенде (Go + Postgres)
📊 Скоринг-оценка 5 независимых осей
Движок на Go рассчитывает текущий статус выживших осей по референсным значениям в БД (с учетом культурных порогов в таблице reference_ranges):
🟢 Стабильно (Green) 🟡 Внимание (Yellow) 🟠 Отклонение (Orange) ⚪️ Нет данных (No Data / Graceful Fallback)
⚡️ Оценка парадокса SDNN (Ось 5):
Зафиксирован утренний RMSSD < 25ms при высоком SDNN > 150ms?
Да (Дисрегуляция вегетативного тонуса)
Бэк на Go игнорирует формальную "норму" анализов и принудительно выставляет Оси 5 статус Orange (🟠), уводя пользователя в Profile 5.
Нет (Штатная оценка)
Показатели HRV оцениваются линейно по стандартным порогам: RMSSD < 25ms → Orange, 25-40ms → Yellow, > 40ms → Green.
Этап 3: Иерархическое профилирование
🧬 Определение метаболического профиля (Шаги 0–4)
Движок прогоняет полученные статусы осей через иерархический фильтр приоритетов. Первый сработавший триггер сверху вниз фиксирует первичный профиль. Остальные отклонения падают в JSONB-модификаторы:
  1. Шаг 0 (Приоритет безопасности): Активирована Ось 5 🟠 (Нейро-вегетативная) или есть симптомы dG/dt → Profile 5
  2. Шаг 1: Активирована Ось 4 🟠 (TSH > 2.5 мМЕ/л) → Profile 4 (Стресс)
  3. Шаг 2: Активирована Ось 3 🟠 (hs-CRP > 1.0 мг/л) → Profile 3 (Воспаление)
  4. Шаг 3: Активирована Ось 1 🟠/🟡 или Ось 2 🟠/🟡 → Profile 2 (Углеводный)
  5. Шаг 4 (Дефолт): Все доступные оси в норме (🟢) → Profile 1 (Метаболическая гибкость)
🔄 Логика пересчета и транзита профилей (День 15+)
🔄 Триггер повторного анализа Baseline vs Follow-up
По истечении 15 дней клинического исследования приложение блокирует генерацию текущей корзины и требует загрузить повторные анализы (Follow-up) и опросники.

Логика транзита на бэкенде (Go):
  1. Бэк сохраняет старый профиль в историю и рассчитывает новые статусы 5 осей.
  2. Вычисляется дельта маркеров: $\Delta \text{HbA1c} = \text{HbA1c}_{\text{follow\_up}} - \text{HbA1c}_{\text{baseline}}$ и $\Delta \text{RMSSD}$.
  3. Транзитная матрица решений движка:
Исходный профиль Клиническая динамика (День 15) Новый профиль Действие бэкенда с корзиной
Profile 5 (Нейро) RMSSD вырос c <25ms до >35ms, симптомы ушли Profile 1 (Гибкость) Снятие ограничений по углеводам, расширение меню, отключение суплементов.
Profile 4 (Стресс) TSH снизился с 2.8 до <2.0 мМЕ/л Profile 1 (Гибкость) Разрешение легких калорийных дефицитов, перевод в поддерживающий режим.
Profile 2 (Углеводный) HbA1c снизился >0.3%, TG/HDL < 1.5 Profile 1 (Гибкость) Постепенное введение сложных углеводов, перевод на фазовое питание.
Любой профиль Динамика отрицательная или отсутствует Без изменений Коррекция Grocery Stability Score, ротация блюд-триггеров, замена белков.
Этап 4: Продуктовая Матрица
🛒 Генерация корзины и расчет Grocery Stability Score
Бэкенд на Go генерирует еженедельную корзину. После покупки чеки сканируются через OCR.

Формула Grocery Stability Score (GSS):
GSS = (Кол-во рекомендованных продуктов в чеке / Общее кол-во продуктов в чеке) * 100 - (Штраф за продукты-триггеры)
  • Штраф за триггеры: Каждая позиция из черного списка профиля (например, печенье для Profile 5) вычитает 10% из скора.
  • GSS > 80% → Отличный комплаенс (стабильная среда).
  • GSS < 50% → Повышенный риск дрейфа и симптомов.
